Q:Tell us about the project you’re proudest of completing (work or personal). What challenge did it address, and what was your role in making it successful?

A: One project I’m most proud of is a retrieval-augmented generation (RAG) backend service I built as a personal project. The goal was to improve how users get context-aware answers from custom data, instead of relying on generic LLM responses.

The main challenge was ensuring the system returned relevant and consistent results. Early on, the retrieval step wasn’t always surfacing the right context, which led to weaker outputs. To address this, I implemented document chunking, tuned top-k retrieval, and added basic reranking and metadata filtering to improve relevance.

My role was end-to-end — I designed the architecture, built the APIs in Go, integrated PostgreSQL with pgvector for vector search, and connected it to LLM APIs for response generation. I also experimented with running models locally using Ollama to explore latency and cost tradeoffs.

What made it successful was iterating quickly based on results and improving the system’s accuracy and flexibility. It gave me hands-on experience with building practical AI systems and understanding how backend design and data quality directly impact outcomes.

Q: Describe a production system you built and deployed end-to-end, and explain the key decisions that guided your approach.

A :One system I built end-to-end was an interview copilot application based on a retrieval-augmented generation (RAG) architecture in Go. The goal was to support context-aware Q&A over custom knowledge sources like resumes, documents, and project data.

From a system design perspective, I structured the pipeline into **three core layers: ingestion, retrieval, and generation**. For ingestion, I implemented document parsing, chunking strategies, and embedding generation. I stored embeddings in pgvector, which I chose for simplicity and tight integration with relational data, while still supporting efficient vector search.

For retrieval, I implemented top-k semantic search, experimented with chunk sizes, and added lightweight reranking to improve relevance. I also considered latency and cost tradeoffs, keeping retrieval efficient to minimize LLM context size.

On the generation side, I integrated LLM APIs with prompt templates designed for consistency and controllability. I exposed the system through Go-based APIs and ensured modularity so embedding models or LLM providers could be swapped easily.

The system was containerized with Docker and deployed with a focus on reliability and extensibility. Overall, I focused on balancing response quality, latency, and system simplicity, while building a foundation that could scale and evolve.
