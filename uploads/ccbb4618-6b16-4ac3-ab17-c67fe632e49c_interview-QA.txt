Q: Describe a specific API you designed or significantly reshaped in production. We're looking for specifics and trade-offs, not general best practices. 
A: One specific API I designed was the backend API for my **AI language learning app**, where the frontend sends a source language, target language, prompt type, difficulty, and topic or phrase, and the backend returns a multiple-choice flashcard generated by an LLM.

The API was consumed by a React/TypeScript frontend. At first, the endpoint was mostly a direct wrapper around the LLM call, but I reshaped it into a more production-style API: the backend checks PostgreSQL first for an existing matching flashcard, returns cached results when possible, and only calls the LLM when needed. It also persists the generated flashcard, answer options, explanation, and generation metadata so results are auditable and reusable.

A meaningful design decision was to make the response contract strict and stable. Since LLM output can be inconsistent, I did not let raw model output flow directly to the frontend. The backend sanitizes the response, validates the JSON shape, stores it in normalized tables, and returns a predictable API response. That made the frontend much simpler and reduced UI failures caused by malformed LLM responses.

The main trade-off was that strict validation sometimes rejects usable-but-imperfect LLM responses, which can increase retries or fallback handling. But I preferred that over allowing inconsistent data into the database or exposing unpredictable responses to the user.

Today, I would improve it by adding explicit API versioning earlier, separating internal generation metadata from the public response model more clearly, and using a formal migration tool instead of schema initialization logic inside the app startup. I would also define typed OpenAPI schemas more strictly from the beginning so frontend and backend contract changes are easier to manage.

Q: Tell us about a non-trivial Postgres (or SQL) problem you've worked through in production — a schema redesign, risky migration, query optimization, indexing or locking issue, or data integrity problem. Briefly: situation, what you did, outcome
A: One non-trivial Postgres problem I worked through was designing the schema for a **Go-based RAG backend** that needed to support multiple embedding providers and models.

The challenge was that different embedding models produce vectors with different dimensions. For example, an OpenAI embedding model might use one vector size, while a local Ollama or open-source embedding model may use a different size. If the schema assumes one fixed `vector(n)` dimension, switching providers can break inserts, retrieval, or indexing.

To solve this, I redesigned the schema around **provider-aware embedding storage**. Instead of treating all embeddings as interchangeable, I stored metadata such as the embedding provider, model name, vector dimension, source type, document ID, chunk ID, and source-level metadata. This made the system safer when switching between OpenAI, Ollama, or other local embedding APIs.

A meaningful design decision was to validate the embedding dimension at startup and before insert, rather than letting bad vectors fail later during retrieval. I also considered different schema options: one shared embeddings table versus separate tables per provider/model. I chose a design that keeps metadata explicit and makes it easier to cleanly filter, delete, or re-ingest knowledge by source and model.

The outcome was a more flexible RAG backend. It could support multiple embedding providers without silently mixing incompatible vectors, and retrieval became more reliable because queries could filter by the correct provider/model/dimension. It also made operational tasks easier, such as re-ingesting documents after changing models or deleting all chunks from a specific source.

What I would improve today is adding more formal versioned migrations and stronger database-level constraints earlier, such as uniqueness constraints on source/chunk/model combinations and clearer migration paths when a model is replaced.

Q: Tell us about a time a system you built or owned failed, misbehaved under load, or had a data integrity issue in production. What happened, how did you diagnose it, and what did you change — in the code, design, or process — to prevent it recurring?
A: One issue I ran into was a data integrity and retrieval bug in my Go-based RAG backend** when I first added support for multiple embedding providers.

- What happened
I initially stored all embeddings in a single table without strictly enforcing metadata consistency. When I started mixing embeddings from different models (e.g., OpenAI vs. local models), I began seeing inconsistent retrieval results—queries would sometimes return irrelevant chunks or fail silently.

Under the hood, the issue was that **vectors with different dimensions and distributions were being mixed together**, and my retrieval queries weren’t filtering strictly by provider/model. In some cases, inserts would succeed but retrieval quality degraded; in others, Postgres would error due to dimension mismatches depending on how the query was constructed.

- How I diagnosed it
I started by logging query inputs and outputs and noticed that irrelevant results often came from documents ingested with a different model. I then inspected the database and realized:

Embeddings from different providers were coexisting without strict separation
There was no enforced contract between “query embedding” and “stored embedding”
Some queries were missing filters on provider/model metadata

I also reproduced the issue by running the same query across different subsets of data and confirmed that mixing embeddings was the root cause.

- What I changed
I made a few key changes:

Schema + metadata redesign: added explicit fields for provider, model, and embedding dimension, and treated them as part of the retrieval contract
Strict filtering in queries: every retrieval query now filters by the exact provider/model used to generate the query embedding
Validation at insert and startup: I added checks to ensure embedding dimensions match expected values before writing to the database
Safer ingestion flow: grouped ingestion by source + model so re-ingestion or deletion can be done cleanly without cross-contamination

- Outcome

Retrieval became consistent and predictable
Eliminated silent quality degradation issues
Made it safe to support multiple embedding providers in the same system
Simplified operational workflows like re-indexing or switching models

- What I’d do differently
I would enforce these constraints earlier at the database level—potentially with stricter table separation or constraints—and introduce better observability (e.g., metrics on retrieval quality or mismatch rates) so issues like this surface faster instead of being discovered through manual debugging.
